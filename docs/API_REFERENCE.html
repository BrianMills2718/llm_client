Package Root Surface

Runtime substrate for multi-agent LLM execution.

Runtime substrate for multi-agent LLM execution.

`llm_client` is the shared public facade for provider transport, agent SDK
routing, prompt rendering, observability, retries/fallbacks, and related
runtime policy. LiteLLM remains the commodity transport backend, but this
package owns the higher-level execution contract that project code relies on.

Usage:
    from llm_client import call_llm, call_llm_structured, call_llm_with_tools, stream_llm

    # Sync
    result = call_llm(
        "gpt-4o",
        [{"role": "user", "content": "Hello"}],
        task="demo",
        trace_id="init/sync",
        max_budget=1.0,
    )
    print(result.content, result.cost)

    # Agent SDK (same interface, different routing)
    result = call_llm(
        "claude-code",
        [{"role": "user", "content": "Fix the bug"}],
        task="demo_agent",
        trace_id="init/agent",
        max_budget=0,
    )
    result = call_llm(
        "claude-code/opus",
        [{"role": "user", "content": "Review code"}],
        task="demo_agent",
        trace_id="init/agent_opus",
        max_budget=0,
    )

    # Batch (concurrent)
    results = call_llm_batch(
        "gpt-4o",
        [msgs1, msgs2, msgs3],
        max_concurrent=5,
        task="demo_batch",
        trace_id="init/batch",
        max_budget=2.0,
    )

    # Streaming
    for chunk in stream_llm(
        "gpt-4o",
        [{"role": "user", "content": "Hello"}],
        task="demo_stream",
        trace_id="init/stream",
        max_budget=1.0,
    ):
        print(chunk, end="")

    # Async
    from llm_client import acall_llm, astream_llm

    result = await acall_llm(
        "gpt-4o",
        [{"role": "user", "content": "Hello"}],
        task="demo_async",
        trace_id="init/async",
        max_budget=1.0,
    )

Legacy difficulty-tier model guidance for task-graph compatibility.

This module maps a coarse difficulty tier to the cheapest currently available
model recommendation for that tier. It remains part of `llm_client` because the
simple `task_graph` runner, analyzer/model-floor logic, and compatibility
imports still depend on it.

New project code should prefer task-based selection via `get_model(task)` and
config fields such as `selection_task`. The difficulty router is advisory, not
authoritative policy: it should not become a second evolving product-facing
model-governance system alongside the shared model registry.

Usage:
    from llm_client.difficulty import get_model_for_difficulty, DifficultyTier

    model = get_model_for_difficulty(2)  # → cheapest available tier-2 model
    model = get_model_for_difficulty(1)  # → local ollama if available, else cloud fallback
    model = get_model_for_difficulty(0)  # → None (scripted, no LLM needed)

LLM client wrapping litellm + agent SDKs.

Fourteen functions (7 sync + 7 async), no class, no mutable state:
- call_llm / acall_llm: basic completion (+ agent SDK routing)
- call_llm_structured / acall_llm_structured: Pydantic extraction (instructor or Responses API)
- call_llm_with_tools / acall_llm_with_tools: tool/function calling
- call_llm_batch / acall_llm_batch: concurrent batch calls
- call_llm_structured_batch / acall_llm_structured_batch: concurrent structured batch
- stream_llm / astream_llm: streaming with retry/fallback
- stream_llm_with_tools / astream_llm_with_tools: streaming with tools

Features:
- Three-tier routing: Agent SDK → Responses API → Chat Completions
- Smart retry with jittered exponential backoff on transient errors,
  empty responses, and JSON parse failures
- Automatic Responses API routing for bare GPT-5 models
  (litellm.responses; when OpenRouter auto-routing is off)
- Agent SDK routing for "claude-code" and "codex" models
- Thinking model detection (Gemini 3/4 → budget_tokens: 0)
- Fallback models — automatic failover to secondary models
- Observability hooks (before_call, after_call, on_error)
- Response caching with sync and async cache protocols
- Fence stripping utility for manual JSON parsing
- Cost tracking via litellm.completion_cost
- finish_reason + raw_response on every result
